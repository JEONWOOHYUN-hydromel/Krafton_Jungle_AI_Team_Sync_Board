# 04. Data Layer

출처: [원본 PDF 1쪽](./full-stack-tech-loadmap.pdf#page=1)

<a id="data-layer"></a>
## Data Layer

데이터 계층은 서비스의 정보를 저장하고 조회하는 영역이다.  
웹 서비스는 결국 데이터를 만들고, 읽고, 바꾸고, 지우는 일을 반복한다.

데이터 계층에서 중요한 질문은 아래와 같다.

```text
어떤 데이터를 저장할 것인가?
어떤 관계가 있는가?
어떻게 빠르게 찾을 것인가?
어떻게 무결성을 지킬 것인가?
장애가 나도 어떻게 안전하게 처리할 것인가?
데이터가 많아지면 어떻게 확장할 것인가?
```

데이터 계층은 서비스의 기억장치다. 화면은 바뀌고 서버는 재시작될 수 있지만, 데이터는 사용자의 행동과 서비스의 상태를 계속 남긴다.  
그래서 DB 키워드는 단순 저장 기술이 아니라 "어떻게 정확히 기억할 것인가", "어떻게 빠르게 찾을 것인가", "동시에 바뀌어도 어떻게 안전하게 유지할 것인가"를 다루는 키워드로 보면 이해가 훨씬 쉽다.

<a id="rdbms"></a>
## RDBMS

RDBMS는 Relational Database Management System, 관계형 데이터베이스 관리 시스템이다.  
데이터를 테이블 형태로 저장하고, 테이블 사이의 관계를 다룬다.

관계형 데이터베이스를 처음 볼 때는 "엑셀처럼 표에 저장하는 것"에서 출발해도 된다.  
다만 엑셀과 다른 점은 테이블 사이의 관계, 제약조건, 트랜잭션, 인덱스를 통해 데이터가 함부로 망가지지 않게 지켜준다는 점이다.

예를 들어 사용자와 게시글은 서로 관계가 있다.

```text
users
- id
- email
- name

posts
- id
- author_id
- title
- content
```

`posts.author_id`는 `users.id`를 참조한다.

아래 상황에서 RDBMS가 잘 맞는다.

- 데이터 구조가 명확하다.
- 관계가 중요하다.
- 트랜잭션이 필요하다.
- 정확성과 무결성이 중요하다.
- SQL로 데이터를 조회하고 싶다.

<a id="postgresql"></a>
## PostgreSQL

PostgreSQL은 강력한 오픈소스 관계형 데이터베이스다.

PostgreSQL은 일반적인 테이블 저장뿐 아니라 JSON, 전문 검색, 확장 기능, 트랜잭션, 인덱스 등 다양한 기능을 제공한다.

아래 상황에서 많이 선택된다.

- 안정적인 RDBMS가 필요하다.
- 복잡한 쿼리와 트랜잭션이 필요하다.
- JSON 데이터도 함께 다루고 싶다.
- 확장 기능을 활용하고 싶다.
- 검색, 지리 정보, vector 검색 같은 확장을 고려한다.

기능이 많기 때문에 처음에는 단순한 테이블, 관계, 인덱스, 트랜잭션부터 이해하는 게 좋다.

<a id="mysql"></a>
## MySQL

MySQL은 널리 사용되는 오픈소스 관계형 데이터베이스다.  
웹 서비스, 커머스, 게시판, CMS 등에서 오래전부터 많이 쓰였다.

아래 상황에서 많이 쓰인다.

- 익숙한 RDBMS가 필요하다.
- 웹 서비스에서 검증된 DB를 쓰고 싶다.
- 기존 시스템이 MySQL 기반이다.
- 읽기/쓰기 패턴이 비교적 일반적이다.

### PostgreSQL과 비교

둘 다 좋은 RDBMS다.  
PostgreSQL은 확장성과 고급 기능이 강하다는 인식이 있고, MySQL은 웹 서비스에서 오래 쓰여 익숙하고 자료가 많다.

<a id="sql"></a>
## SQL

SQL은 관계형 데이터베이스와 대화하기 위한 언어다.

SQL로 데이터를 조회, 생성, 수정, 삭제한다.

```sql
SELECT * FROM posts WHERE author_id = 1;
```

RDBMS를 쓰면 SQL은 필수다.  
ORM을 쓰더라도 SQL의 기본을 알아야 성능 문제와 데이터 구조를 이해할 수 있다.

ORM은 SQL을 덜 쓰게 해주는 도구이지, SQL을 몰라도 되게 해주는 도구는 아니다.  
느린 쿼리, 잘못된 Join, 인덱스 누락 같은 문제는 결국 SQL 관점으로 봐야 잡힌다.

<a id="crud"></a>
## CRUD

PDF에는 `CURD`라고 적혀 있지만 일반적으로는 `CRUD`라고 쓴다.  
CRUD는 Create, Read, Update, Delete의 약자다.

대부분의 서비스 기능은 CRUD에서 출발한다.

```text
Create: 새 게시글 작성
Read: 게시글 조회
Update: 게시글 수정
Delete: 게시글 삭제
```

거의 모든 데이터 기반 서비스에서 사용한다.  
사용자, 게시글, 댓글, 상품, 주문, 파일 등 대부분의 리소스는 CRUD API를 가진다.

CRUD는 쉬워 보이지만 권한, 검증, 트랜잭션, 삭제 정책까지 고려하면 단순하지 않다.  
예를 들어 삭제는 실제로 DB에서 지우는 hard delete와 삭제 표시만 하는 soft delete가 있다.

<a id="join"></a>
## Join

Join은 여러 테이블의 데이터를 관계를 기준으로 합쳐서 조회하는 것이다.

게시글과 작성자 정보를 함께 보고 싶다면 Join이 필요하다.

```sql
SELECT posts.title, users.name
FROM posts
JOIN users ON posts.author_id = users.id;
```

아래 상황에서 사용한다.

- 게시글과 작성자를 함께 조회한다.
- 주문과 상품 정보를 함께 조회한다.
- 댓글과 사용자 정보를 함께 조회한다.
- N:M 관계를 연결 테이블로 조회한다.

### Inner Join, Outer Join

Inner Join은 양쪽에 모두 매칭되는 데이터만 가져온다.  
Outer Join은 매칭이 없어도 한쪽 데이터를 유지한다.

```text
Inner Join: 작성자가 있는 게시글만
Left Join: 작성자 정보가 없어도 게시글은 유지
```

<a id="data-modeling"></a>
## Data Modeling

데이터 모델링은 서비스에 필요한 데이터를 어떤 테이블과 관계로 표현할지 설계하는 일이다.

예를 들어 게시판을 만들면 아래 질문을 해야 한다.

```text
사용자는 어떤 정보를 가지는가?
게시글은 작성자와 어떤 관계인가?
댓글은 게시글과 어떤 관계인가?
태그는 게시글과 N:M 관계인가?
삭제된 사용자의 게시글은 어떻게 처리할 것인가?
```

DB 설계는 나중에 바꾸기 어렵다.  
처음부터 완벽할 필요는 없지만, 핵심 관계와 제약은 신중하게 잡아야 한다.

<a id="erd"></a>
## ERD

ERD는 Entity Relationship Diagram이다.  
엔티티와 관계를 그림으로 표현한 것이다.

ERD는 테이블, 컬럼, 관계를 시각화한다.

```text
User 1:N Post
Post 1:N Comment
Post N:M Tag
```

### 관계 종류

1:1은 하나가 하나와 연결된다.  
1:N은 하나가 여러 개와 연결된다.  
N:M은 여러 개가 여러 개와 연결된다.

DB를 설계하거나 팀원에게 데이터 구조를 설명할 때 사용한다.

<a id="primary-key-foreign-key-index"></a>
## Primary Key, Foreign Key, Index

### Primary Key

Primary Key는 테이블에서 각 row를 고유하게 식별하는 값이다.

```text
users.id
posts.id
```

Primary Key는 중복될 수 없고 비어 있으면 안 된다.

### Foreign Key

Foreign Key는 다른 테이블의 Primary Key를 참조하는 값이다.

```text
posts.author_id -> users.id
```

Foreign Key는 데이터 관계와 무결성을 지키는 데 사용된다.

### Index

Index는 데이터를 빠르게 찾기 위한 구조다.

```sql
CREATE INDEX idx_posts_author_id ON posts(author_id);
```

Primary Key와 Foreign Key는 데이터 관계를 지키는 데 중요하고, Index는 조회 성능에 중요하다.

Index는 읽기를 빠르게 하지만 쓰기에는 비용이 생긴다.  
모든 컬럼에 인덱스를 붙이면 오히려 비효율적일 수 있다.

<a id="normalization"></a>
## Normalization

정규화는 데이터 중복을 줄이고 무결성을 높이기 위해 테이블을 잘 나누는 과정이다.

예를 들어 사용자 이름을 게시글 테이블마다 저장하면 사용자가 이름을 바꿀 때 모든 게시글을 수정해야 한다.  
정규화하면 사용자 정보는 users 테이블에 저장하고, 게시글은 author_id만 가진다.

관계형 DB 설계의 기본이다.  
데이터 중복이 많거나 수정 시 여러 곳을 바꿔야 한다면 정규화를 고려한다.

정규화를 너무 많이 하면 조회할 때 Join이 많아질 수 있다.  
실무에서는 정규화와 성능 사이에서 균형을 잡는다.

<a id="transaction"></a>
## Transaction

트랜잭션은 여러 DB 작업을 하나의 작업 단위로 묶는 것이다.

예를 들어 계좌 이체는 두 작업이 함께 성공해야 한다.

```text
A 계좌에서 1000원 차감
B 계좌에 1000원 추가
```

하나만 성공하면 데이터가 망가진다.  
트랜잭션은 모두 성공하면 commit, 하나라도 실패하면 rollback한다.

아래 상황에서 중요하다.

- 결제
- 주문
- 포인트 적립/차감
- 재고 변경
- 여러 테이블을 함께 수정하는 작업

<a id="isolation"></a>
## Isolation

Isolation은 트랜잭션 격리 수준이다.  
동시에 여러 트랜잭션이 실행될 때 서로 얼마나 영향을 주지 않게 할지 정한다.

격리 수준이 낮으면 성능은 좋아질 수 있지만 이상한 읽기가 발생할 수 있다.  
격리 수준이 높으면 안전하지만 락이 많아지고 성능이 떨어질 수 있다.

### 대표 문제

- Dirty Read: commit되지 않은 데이터를 읽음
- Non-repeatable Read: 같은 데이터를 두 번 읽었는데 값이 달라짐
- Phantom Read: 같은 조건으로 조회했는데 row 개수가 달라짐

동시성이 많은 서비스에서 중요하다.  
결제, 재고, 좌석 예약처럼 동시에 같은 데이터를 수정하는 기능에서는 특히 중요하다.

<a id="lock-deadlock"></a>
## Lock/Deadlock

Lock은 동시에 같은 데이터를 수정하지 못하게 잠그는 것이다.  
Deadlock은 서로가 서로의 lock을 기다리면서 멈춰버리는 상황이다.

### Lock

예를 들어 한 사용자가 재고를 수정하는 동안 다른 사용자가 같은 재고를 동시에 바꾸면 충돌이 생길 수 있다.  
Lock은 이런 충돌을 막는다.

### Deadlock

두 트랜잭션이 서로 다른 순서로 lock을 잡으면 deadlock이 생길 수 있다.

```text
T1: A lock 획득 -> B lock 기다림
T2: B lock 획득 -> A lock 기다림
```

동시에 많은 사용자가 같은 데이터를 수정하는 서비스에서 중요하다.

트랜잭션을 짧게 유지하고, lock을 잡는 순서를 일관되게 만드는 것이 중요하다.

<a id="reliability"></a>
## Reliability

신뢰성은 시스템이 실패 상황에서도 예측 가능하게 동작하는 능력이다.  
서버, DB, 외부 API, 네트워크는 언제든 실패할 수 있다.

신뢰성을 높이는 대표 개념이 멱등성, 재시도, 타임아웃, 회로차단기다.

이 네 가지는 같이 묶어서 봐야 한다.  
타임아웃은 무한 대기를 막고, 재시도는 일시적 실패를 다시 처리하고, 멱등성은 재시도해도 중복 처리되지 않게 하고, 회로차단기는 계속 실패하는 곳으로 요청을 몰아넣지 않게 한다.

<a id="idempotency"></a>
## Idempotency

멱등성은 같은 요청을 여러 번 보내도 결과가 한 번 보낸 것과 같다는 성질이다.

예를 들어 `DELETE /posts/1`은 여러 번 호출해도 결과적으로 게시글이 삭제된 상태라면 멱등적이다.  
반면 `POST /payments`는 여러 번 호출하면 결제가 여러 번 될 수 있으므로 위험하다.

아래 상황에서 중요하다.

- 결제
- 주문 생성
- 포인트 적립
- 외부 API 재시도
- 네트워크 실패 후 같은 요청을 다시 보낼 수 있는 기능

중요한 생성 작업에는 idempotency key를 사용하기도 한다.  
같은 key로 들어온 요청은 한 번만 처리되게 만드는 방식이다.

<a id="retry"></a>
## Retry

재시도는 실패한 작업을 다시 시도하는 것이다.

네트워크 오류나 일시적인 외부 API 장애는 다시 시도하면 성공할 수 있다.

아래 상황에서 사용한다.

- 일시적인 네트워크 장애
- 외부 API 503
- 타임아웃
- 메시지 큐 작업 실패

무조건 재시도하면 장애를 더 키울 수 있다.  
재시도 횟수, 간격, backoff, 멱등성을 함께 고려해야 한다.

<a id="timeout"></a>
## Timeout

타임아웃은 작업을 무한히 기다리지 않고 일정 시간이 지나면 실패로 처리하는 것이다.

외부 API가 응답하지 않으면 서버가 계속 기다릴 수 없다.  
타임아웃을 설정해서 일정 시간 후 요청을 중단한다.

거의 모든 외부 통신에 필요하다.

- HTTP API 호출
- DB 연결
- 메시지 처리
- 파일 업로드/다운로드

너무 짧으면 정상 요청도 실패하고, 너무 길면 장애 시 자원이 묶인다.

<a id="circuit-breaker"></a>
## Circuit Breaker

회로차단기는 계속 실패하는 외부 시스템에 요청을 잠시 보내지 않도록 막는 패턴이다.

외부 API가 장애 상태인데 계속 요청을 보내면 우리 서버도 느려지고 장애가 전파된다.  
회로차단기는 실패가 많아지면 잠시 차단하고, 일정 시간이 지나면 다시 테스트한다.

아래 상황에서 사용한다.

- 외부 결제 API
- 외부 인증 API
- AI API
- 다른 마이크로서비스 호출

회로차단기는 실패를 없애는 것이 아니라 장애 전파를 줄이는 장치다.  
차단 중 사용자에게 어떤 fallback을 줄지도 설계해야 한다.

<a id="scalability"></a>
## Scalability

확장성은 사용량이 늘어났을 때 시스템이 감당할 수 있게 커지는 능력이다.

확장에는 두 가지 방향이 있다.

```text
Scale up: 더 좋은 서버로 키운다.
Scale out: 서버 수를 늘린다.
```

DB에서는 partition과 sharding이 확장성 키워드로 자주 나온다.

Partition과 Sharding은 둘 다 "나눈다"는 느낌이 있지만 난이도가 다르다.  
Partition은 보통 한 DB 안에서 큰 테이블을 관리하기 좋게 나누는 쪽이고, Sharding은 여러 DB 서버로 데이터를 흩어 저장하는 쪽이다. 그래서 Sharding은 훨씬 강력하지만 운영 부담도 훨씬 크다.

<a id="partition"></a>
## Partition

Partition은 큰 테이블을 더 작은 단위로 나누는 것이다.  
보통 같은 DB 안에서 테이블을 논리적/물리적으로 나눈다.

예를 들어 로그 테이블을 날짜별로 나눌 수 있다.

```text
logs_2026_01
logs_2026_02
logs_2026_03
```

또는 PostgreSQL의 partition table을 사용할 수 있다.

아래 상황에서 사용한다.

- 테이블이 매우 커진다.
- 날짜별 조회가 많다.
- 오래된 데이터를 쉽게 보관/삭제하고 싶다.
- 특정 범위만 빠르게 조회하고 싶다.

<a id="sharding"></a>
## Sharding

Sharding은 데이터를 여러 DB 서버에 나누어 저장하는 것이다.

예를 들어 user_id 기준으로 데이터를 여러 DB에 나눈다.

```text
user_id 1-100000     -> shard 1
user_id 100001-200000 -> shard 2
```

아래 상황에서 사용한다.

- 하나의 DB 서버로 감당할 수 없을 만큼 데이터가 크다.
- 쓰기 부하가 너무 크다.
- 수평 확장이 필요하다.

Sharding은 난이도가 높다.  
데이터 이동, cross-shard query, transaction, 운영 복잡도가 크게 증가한다.  
초기 서비스에서는 보통 인덱스, 캐싱, read replica, partition을 먼저 고려한다.
