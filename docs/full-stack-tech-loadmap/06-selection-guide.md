# 06. 한눈에 보는 선택 기준

출처: [원본 PDF 1쪽](./full-stack-tech-loadmap.pdf#page=1)

이 문서는 앞의 기술 설명을 빠르게 비교하기 위한 요약이다.  
기술 선택에서 중요한 건 "더 유명한 것"이 아니라 "내가 풀 문제에 맞는 것"이다.

표는 정답표가 아니다. 오히려 팀에서 대화할 때 쓰는 기준표에 가깝다.  
예를 들어 REST와 GraphQL 중 하나를 고르는 문제는 "뭐가 더 최신인가"가 아니라 "우리 화면이 필요한 데이터 모양이 얼마나 자주 달라지는가"의 문제다. Redis를 쓸지 말지도 "좋은 기술이라서"가 아니라 "반복 조회나 세션, rate limit처럼 빠른 임시 저장소가 필요한가"를 보고 정해야 한다.

## 언어 선택

| 기술 | 의미 | 주로 쓰는 곳 | 선택하면 좋은 상황 |
| --- | --- | --- | --- |
| TypeScript | 타입이 있는 JavaScript | 프론트엔드, Node.js 백엔드 | UI와 API 데이터 구조를 안전하게 다루고 싶을 때 |
| Java | 안정적인 정적 타입 언어 | 기업용 백엔드, Spring Boot | 장기 운영, 대규모 팀, 안정성이 중요할 때 |
| Python | 간결하고 생태계가 큰 언어 | API 서버, 데이터, AI | 빠른 구현, AI/데이터 라이브러리 활용이 중요할 때 |

## 프론트엔드 선택

| 기술 | 의미 | 주로 쓰는 곳 | 선택하면 좋은 상황 |
| --- | --- | --- | --- |
| React | 컴포넌트 기반 UI 라이브러리 | 대부분의 현대 웹 앱 | 생태계가 크고 유연한 UI 도구가 필요할 때 |
| Vue.js | 템플릿 중심 프론트엔드 프레임워크 | 웹 앱, 관리자 페이지 | 입문 난이도와 공식 생태계를 중시할 때 |
| CSR | 브라우저에서 화면 렌더링 | 대시보드, 로그인 후 앱 | SEO보다 상호작용과 구조 단순성이 중요할 때 |
| SSR | 서버에서 HTML 렌더링 | 공개 웹, 커머스 | 첫 로딩과 SEO가 중요할 때 |
| SSG | 빌드 시 HTML 생성 | 문서, 블로그 | 콘텐츠가 자주 바뀌지 않을 때 |
| ISR | 정적 페이지를 주기적으로 갱신 | 상품 상세, 콘텐츠 페이지 | 빠른 응답과 주기적 갱신이 모두 필요할 때 |
| RSC | React 컴포넌트를 서버에서 실행 | Next.js 계열 | 서버 데이터 조회와 클라이언트 번들 최적화가 필요할 때 |

## 상태 관리 선택

| 기술 | 의미 | 선택하면 좋은 상황 |
| --- | --- | --- |
| Local State | 컴포넌트 내부 상태 | 해당 컴포넌트 안에서만 필요한 값 |

| Redux Toolkit | 구조화된 전역 상태 관리 | 전역 상태가 크고 변경 흐름 추적이 중요할 때 |
| Zustand | 가벼운 전역 상태 관리 | 간단한 전역 상태를 빠르게 관리하고 싶을 때 |
| Pinia | Vue용 상태 관리 | Vue 프로젝트에서 전역 상태가 필요할 때 |

## API 선택

| 기술 | 의미 | 주로 쓰는 곳 | 선택하면 좋은 상황 |
| --- | --- | --- | --- |
| REST | 리소스 중심 HTTP API | 일반 웹 서비스 | CRUD, 인증, 검색 등 명확한 리소스 API가 많을 때 |
| GraphQL | 클라이언트가 응답 모양을 정하는 API | 복잡한 클라이언트/여러 앱 | 화면마다 필요한 데이터 조합이 많이 다를 때 |
| gRPC | 고성능 RPC 통신 | 서버 간 통신 | 내부 서비스 통신, 고성능, 명확한 protobuf 계약이 필요할 때 |

| OAuth2 | 외부 서비스 권한 위임 | Google/GitHub/Slack 등 외부 권한을 받아야 할 때 |
| OIDC | OAuth2 위의 인증 표준 | 외부 인증 제공자로 로그인할 때 |

## 비동기 처리 선택

| 기술 | 의미 | 선택하면 좋은 상황 |

## 데이터베이스 선택

| 기술 | 의미 | 선택하면 좋은 상황 |
| --- | --- | --- |
| RDBMS | 관계형 데이터베이스 | 데이터 관계와 무결성이 중요할 때 |
| PostgreSQL | 확장성이 강한 오픈소스 RDBMS | 복잡한 쿼리, 확장 기능, 안정성이 필요할 때 |
| MySQL | 널리 쓰이는 오픈소스 RDBMS | 일반 웹 서비스, 익숙한 운영 환경 |

## 데이터 설계 선택

| 기술 | 의미 | 언제 중요한가 |
| --- | --- | --- |
| CRUD | 생성/조회/수정/삭제 | 대부분의 데이터 기반 기능 |
| Join | 여러 테이블 결합 조회 | 관계 데이터를 함께 보여줄 때 |
| ERD | 데이터 관계 다이어그램 | DB 구조를 설계하고 공유할 때 |
| Primary Key | row 고유 식별자 | 모든 테이블 설계의 기본 |
| Foreign Key | 다른 테이블 참조 | 관계와 무결성을 지킬 때 |
| Index | 조회 성능 개선 | 검색 조건, 정렬, join이 잦은 컬럼 |
| Normalization | 중복 제거와 무결성 향상 | 데이터 중복과 수정 오류를 줄이고 싶을 때 |

## 안정성 선택

| 기술 | 의미 | 선택하면 좋은 상황 |
| --- | --- | --- |
| Transaction | 여러 DB 작업을 하나로 묶음 | 결제, 주문, 포인트, 재고 |
| Isolation | 동시 트랜잭션 간 격리 | 동시에 같은 데이터를 다루는 기능 |
| Lock | 동시 수정을 막는 잠금 | 충돌 방지가 필요할 때 |
| Deadlock | 서로 lock을 기다리며 멈춤 | lock 순서와 트랜잭션 길이를 관리해야 할 때 |
| Idempotency | 여러 번 실행해도 결과가 같음 | 재시도 가능한 요청, 결제/주문 |
| Retry | 실패한 작업 재시도 | 일시적 장애 |
| Timeout | 무한 대기 방지 | 외부 API, DB, 네트워크 호출 |
| Circuit Breaker | 장애 시스템 호출 차단 | 외부 API 장애 전파 방지 |

## 확장성 선택

| 기술 | 의미 | 선택하면 좋은 상황 |
| --- | --- | --- |
| Caching | 자주 쓰는 데이터를 빠른 저장소에 둠 | 반복 조회, 외부 API 비용 감소 |
| Redis | 빠른 메모리 저장소 | 캐시, 세션, rate limit, queue |
| Partition | 큰 테이블을 나눔 | 날짜/범위별 대용량 테이블 |
| Sharding | 여러 DB 서버에 데이터 분산 | 단일 DB로 감당할 수 없을 때 |

## 아키텍처 선택

| 기술 | 의미 | 선택하면 좋은 상황 |
| --- | --- | --- |
| MVC | Model/View/Controller 분리 | 기본적인 웹 앱 구조 |
| Layered Architecture | Controller/Service/Repository 계층 분리 | API 서버를 명확하게 나누고 싶을 때 |
| Modular Monolith | 하나의 앱 안에서 모듈 분리 | 마이크로서비스 전 단계, 기능별 경계가 필요할 때 |
| Clean Architecture | 핵심 로직을 외부 기술에서 보호 | 장기 유지보수와 테스트가 중요할 때 |
| Hexagonal Architecture | Port/Adapter로 외부 연결 분리 | DB/API 교체와 테스트 대체가 중요할 때 |
| ADR | 기술 결정 기록 | 왜 이 선택을 했는지 팀에 남기고 싶을 때 |
| Microservices | 여러 독립 서비스로 분리 | 조직/서비스 규모가 크고 독립 배포가 필요할 때 |
| Event-driven Architecture | 이벤트 발행/구독으로 처리 | 후속 작업이 많고 결합도를 낮추고 싶을 때 |
| CQRS | 읽기와 쓰기 모델 분리 | 읽기/쓰기 요구사항이 크게 다를 때 |

## 내가 이해한 큰 그림

이 로드맵은 사실 기술 이름 목록이 아니라 웹 서비스가 커질 때 생기는 문제의 목록에 가깝다.

```text
화면을 어떻게 만들지 -> Frontend
요청을 어떻게 받을지 -> Backend/API
누가 요청했는지 어떻게 알지 -> Authentication
데이터를 어디에 저장하지 -> Data Layer
동시에 수정하면 어떻게 하지 -> Transaction/Lock
느려지면 어떻게 하지 -> Cache/Scaling
장애가 나면 어떻게 버티지 -> Reliability
코드가 커지면 어떻게 나누지 -> Architecture
```

그래서 기술을 공부할 때는 이름을 외우기보다, 각 기술이 해결하려는 질문을 같이 기억하는 게 좋다.
