# 02. Frontend

출처: [원본 PDF 1쪽](./full-stack-tech-loadmap.pdf#page=1)

<a id="frontend"></a>
## Frontend

프론트엔드는 사용자가 직접 보고 조작하는 영역이다.  
버튼, 입력창, 게시글 목록, 로그인 화면, 대시보드, 채팅창처럼 브라우저에서 보이는 거의 모든 것이 프론트엔드에 속한다.

프론트엔드의 핵심은 단순히 예쁜 화면을 만드는 것이 아니다.

```text
사용자의 행동
-> 화면 상태 변화
-> 서버 API 호출
-> 응답 반영
-> 에러와 로딩 상태 처리
```

이 흐름을 자연스럽게 만드는 것이 프론트엔드의 역할이다.

프론트엔드를 처음 공부할 때는 기술이 너무 많이 보인다. React, Vue, 라우팅, 상태관리, 렌더링, UI 프레임워크가 전부 따로 노는 것처럼 보인다.  
하지만 결국은 하나의 질문으로 모인다. "사용자가 어떤 상태를 보고 있고, 어떤 행동을 했을 때 화면이 어떻게 바뀌어야 하는가?" 이 질문에 답하기 위해 컴포넌트, state, props, event, routing 같은 도구가 붙는다.

<a id="html-css-javascript-typescript"></a>
## HTML/CSS/JavaScript/TypeScript

프론트엔드의 가장 기본이 되는 네 가지다.

### HTML

HTML은 문서의 구조를 만든다.  
제목, 문단, 버튼, 폼, 이미지, 링크 같은 요소를 표현한다.

```html
<h1>게시글 목록</h1>
<button>글쓰기</button>
```

HTML은 "무엇이 있는가"를 표현한다.

### CSS

CSS는 화면의 모양을 만든다.  
색상, 크기, 간격, 배치, 반응형 레이아웃을 담당한다.

```css
.button {
  padding: 8px 12px;
  background: black;
  color: white;
}
```

CSS는 "어떻게 보이는가"를 표현한다.

### JavaScript

JavaScript는 화면에 동작을 넣는다.  
클릭 이벤트, 입력값 처리, API 호출, 동적 렌더링을 담당한다.

```js
button.addEventListener("click", () => {
  alert("clicked");
});
```

JavaScript는 "어떻게 움직이는가"를 표현한다.

### TypeScript

TypeScript는 JavaScript에 타입을 붙인다.  
프론트엔드 앱이 커질수록 API 응답, 컴포넌트 props, 상태 구조가 복잡해지는데, TypeScript는 이 구조를 명확하게 만든다.

```ts
type Post = {
  id: number;
  title: string;
  content: string;
};
```

TypeScript는 "어떤 데이터가 오고 가는가"를 안전하게 표현한다.

<a id="frontend-framework"></a>
## Frontend Framework

프론트엔드 프레임워크는 화면을 더 구조적으로 만들기 위한 도구다.  
HTML/CSS/JavaScript만으로도 웹페이지를 만들 수 있지만, 화면이 복잡해지면 상태 관리, 재사용, 라우팅, 렌더링을 직접 관리하기 어렵다.

그래서 React, Vue 같은 프레임워크 또는 라이브러리를 쓴다.

<a id="react"></a>
## React

React는 UI를 컴포넌트 단위로 만드는 JavaScript 라이브러리다.

React는 화면을 작은 컴포넌트로 쪼갠다.

```tsx
function PostCard({ title }: { title: string }) {
  return <article>{title}</article>;
}
```

이렇게 하면 게시글 카드, 버튼, 입력창, 모달 같은 UI 조각을 재사용할 수 있다.

React는 아래 상황에서 많이 쓴다.

- 복잡한 사용자 인터페이스를 만든다.
- 컴포넌트를 재사용하고 싶다.
- 상태에 따라 화면이 자주 바뀐다.
- 생태계가 큰 프론트엔드 기술을 쓰고 싶다.

### 초보자가 헷갈리기 쉬운 점

React는 프레임워크라기보다 UI 라이브러리에 가깝다.  
라우팅, 상태 관리, 데이터 fetching, 빌드 도구는 별도로 선택해야 한다. 그래서 React 프로젝트에는 보통 Vite, React Router, TanStack Query 같은 도구가 함께 붙는다.

요약하면 React는 "상태에 따라 UI를 다시 그리는 컴포넌트 기반 도구"다.

<a id="vuejs"></a>
## Vue.js

Vue.js는 React처럼 UI를 만드는 프론트엔드 프레임워크다.  
템플릿 문법이 직관적이고, 공식 생태계가 잘 정리되어 있어서 입문하기 쉽다는 장점이 있다.

Vue는 HTML에 가까운 템플릿과 JavaScript 로직을 결합해서 컴포넌트를 만든다.

```vue
<template>
  <button @click="count++">{{ count }}</button>
</template>
```

아래 상황에서 Vue가 잘 맞는다.

- 팀이 HTML 템플릿 방식에 익숙하다.
- 프레임워크가 제공하는 공식 도구 세트를 선호한다.
- 빠르게 프론트엔드 앱을 만들고 싶다.

### React와의 차이

React는 JavaScript/TypeScript 안에서 UI를 표현하는 느낌이 강하고, Vue는 HTML 템플릿 중심으로 UI를 표현하는 느낌이 강하다.  
둘 다 좋은 선택이고, 중요한 건 팀의 익숙함과 생태계 선택이다.

<a id="rendering"></a>
## Rendering

렌더링은 데이터를 화면으로 그리는 과정이다.  
웹에서는 "HTML을 어디에서 만들 것인가"가 큰 기준이 된다.

```text
브라우저에서 만들면 CSR
서버에서 만들면 SSR
미리 만들어두면 SSG
미리 만들고 나중에 갱신하면 ISR
서버 컴포넌트 단위로 다루면 RSC
```

이 다섯 개는 서로 완전히 다른 세계라기보다, "화면을 언제, 어디서 준비할까?"에 대한 선택지다.  
로그인 후 대시보드처럼 검색 노출이 중요하지 않고 상호작용이 많으면 CSR이 편하다. 반대로 블로그, 커머스 상품 페이지, 공개 문서처럼 첫 화면과 검색 노출이 중요하면 SSR이나 SSG 쪽을 보게 된다. ISR은 정적 페이지의 빠름을 유지하면서 가끔 새로 고치고 싶을 때, RSC는 React 컴포넌트 자체를 서버와 클라이언트로 나눠 최적화하고 싶을 때 등장한다.

<a id="csr"></a>
## CSR

CSR은 Client-Side Rendering이다.  
브라우저가 JavaScript를 내려받은 뒤, 클라이언트에서 화면을 그린다.

처음에는 거의 빈 HTML을 받고, React 같은 앱이 브라우저에서 실행되면서 화면을 만든다.

```text
브라우저 접속
-> JS 다운로드
-> React 실행
-> API 호출
-> 화면 렌더링
```

아래 상황에서 CSR이 잘 맞는다.

- 로그인 후 사용하는 내부 서비스
- 대시보드
- 관리자 페이지
- SEO가 크게 중요하지 않은 앱
- 사용자 상호작용이 많은 앱

처음 로딩이 느려질 수 있고, 검색엔진 최적화에 불리할 수 있다.  
하지만 구조가 단순하고 프론트와 백엔드를 분리하기 쉽다.

<a id="ssr"></a>
## SSR

SSR은 Server-Side Rendering이다.  
서버가 HTML을 먼저 만들어서 브라우저에 보내고, 브라우저는 받은 HTML을 바로 보여준다.

```text
브라우저 요청
-> 서버에서 데이터 조회
-> 서버가 HTML 생성
-> 브라우저에 완성된 HTML 전달
```

아래 상황에서 SSR이 많이 쓰인다.

- 첫 화면 로딩 속도가 중요하다.
- SEO가 중요하다.
- 공개 페이지가 많다.
- 사용자마다 다른 데이터를 서버에서 바로 보여줘야 한다.

서버가 HTML 생성까지 담당하므로 서버 부담이 늘 수 있다.  
또 프론트엔드와 백엔드 역할이 섞일 수 있어서 구조를 잘 잡아야 한다.

<a id="ssg"></a>
## SSG

SSG는 Static Site Generation이다.  
빌드 시점에 HTML을 미리 만들어두는 방식이다.

```text
빌드할 때 HTML 생성
-> CDN에 배포
-> 사용자는 이미 만들어진 HTML을 받음
```

아래 상황에서 좋다.

- 블로그
- 문서 사이트
- 마케팅 페이지
- 자주 바뀌지 않는 콘텐츠

데이터가 자주 바뀌면 매번 다시 빌드해야 하므로 불편할 수 있다.

<a id="isr"></a>
## ISR

ISR은 Incremental Static Regeneration이다.  
정적 페이지를 미리 만들되, 일정 시간이 지나면 다시 생성해서 갱신하는 방식이다.

SSG와 SSR의 중간 느낌이다.  
처음에는 정적 페이지처럼 빠르게 제공하고, 시간이 지나면 백그라운드에서 새 HTML을 만든다.

아래 상황에서 좋다.

- 콘텐츠가 바뀌지만 실시간일 필요는 없다.
- 빠른 응답과 주기적 갱신을 동시에 원한다.
- 상품 상세, 블로그, 뉴스성 페이지처럼 캐싱이 중요한 페이지

실시간 데이터에는 적합하지 않다. 사용자가 약간 오래된 데이터를 볼 수 있다.

<a id="rsc"></a>
## RSC

RSC는 React Server Components이다.  
React 컴포넌트 중 일부를 서버에서 실행할 수 있게 하는 개념이다.

서버에서만 필요한 데이터 조회나 무거운 로직을 클라이언트 번들에 포함하지 않고 처리할 수 있다.

Next.js 같은 최신 React 프레임워크에서 많이 다룬다.  
서버 렌더링과 클라이언트 상호작용을 섞어서 최적화하고 싶을 때 쓰인다.

### 초보자가 헷갈리기 쉬운 점

RSC는 단순히 SSR과 같은 말이 아니다.  
SSR은 HTML을 서버에서 만드는 방식이고, RSC는 React 컴포넌트 자체를 서버/클라이언트로 나눠 다루는 방식이다.

<a id="component-design"></a>
## Component Design

컴포넌트 설계는 UI를 어떤 단위로 나눌지 정하는 것이다.

좋은 컴포넌트는 한 가지 역할을 명확하게 가진다.

```text
PostList
PostCard
PostEditor
CommentList
CommentForm
```

이렇게 나누면 재사용, 테스트, 수정이 쉬워진다.

화면이 커질수록 중요하다.  
처음에는 한 파일에 다 넣어도 되지만, 기능이 늘어나면 컴포넌트 경계가 없어서 수정이 어려워진다.

처음부터 너무 잘게 쪼개면 오히려 복잡해진다.  
반복되는 UI, 독립적인 역할, 상태 경계가 보일 때 컴포넌트로 나누는 게 좋다.

<a id="state-props"></a>
## State/Props

State와 Props는 React/Vue 같은 컴포넌트 기반 UI에서 핵심 개념이다.

둘을 구분하는 감각은 단순하다. Props는 "밖에서 들어오는 값"이고, State는 "컴포넌트가 직접 기억하는 값"이다.  
게시글 카드가 부모로부터 `title`을 받는다면 그건 props이고, 댓글 입력창에 사용자가 타이핑 중인 값은 state에 가깝다.

### Props

Props는 부모 컴포넌트가 자식 컴포넌트에게 전달하는 값이다.

```tsx
<PostCard title="첫 번째 글" />
```

Props는 컴포넌트를 재사용 가능하게 만든다.

### State

State는 컴포넌트가 내부적으로 기억하는 값이다.

```tsx
const [count, setCount] = useState(0);
```

State가 바뀌면 화면도 다시 렌더링된다.

입력값, 모달 열림 여부, 선택된 탭, API 응답 데이터처럼 화면이 바뀌는 모든 곳에서 중요하다.

State를 너무 많이 흩뿌리면 데이터 흐름이 복잡해진다.  
가까운 컴포넌트 안에서만 필요한 값은 local state로 두고, 여러 화면에서 공유해야 하는 값은 상태 관리 도구를 고려한다.

<a id="event"></a>
## Event

이벤트는 사용자의 행동이나 브라우저에서 발생하는 신호다.

클릭, 입력, 스크롤, 키보드 입력, 폼 제출 등이 이벤트다.

```tsx
<button onClick={() => alert("clicked")}>저장</button>
```

프론트엔드는 사용자의 행동에 반응하는 영역이므로 이벤트 처리는 거의 모든 기능에서 필요하다.

이벤트 핸들러 안에 너무 많은 로직을 넣으면 테스트와 유지보수가 어려워진다.  
핸들러는 입력을 받아 적절한 함수나 API 호출로 넘기는 정도로 유지하는 게 좋다.

<a id="life-cycle"></a>
## Life Cycle

라이프 사이클은 컴포넌트가 생성되고, 업데이트되고, 사라지는 흐름이다.

React에서는 `useEffect` 같은 도구로 특정 시점의 작업을 처리한다.

```tsx
useEffect(() => {
  fetchPosts();
}, []);
```

아래 상황에서 중요하다.

- 화면이 처음 열릴 때 데이터를 가져온다.
- 특정 값이 바뀔 때 다시 계산한다.
- 컴포넌트가 사라질 때 구독을 해제한다.

라이프 사이클을 잘못 다루면 무한 API 호출, 메모리 누수, 오래된 상태 참조가 생길 수 있다.  
요즘은 서버 데이터 fetching은 TanStack Query 같은 도구에 맡기는 경우가 많다.

<a id="routing"></a>
## Routing

라우팅은 URL에 따라 어떤 화면을 보여줄지 정하는 것이다.

```text
/login       -> 로그인 화면
/posts       -> 게시글 목록
/posts/1     -> 게시글 상세
/settings    -> 설정 화면
```

프론트엔드 라우팅은 브라우저 안에서 화면을 바꾼다.  
서버 라우팅은 서버가 요청 URL에 따라 응답을 다르게 준다.

여러 페이지가 있는 웹 앱에서는 필수다.

라우팅은 단순 화면 전환이 아니라 인증과도 연결된다.  
예를 들어 로그인하지 않은 사용자는 `/settings`에 접근하지 못하게 해야 한다.

<a id="state-management"></a>
## State Management

상태 관리는 앱 전체에서 사용되는 값을 어디에 저장하고 어떻게 변경할지 정하는 것이다.

상태에는 여러 종류가 있고, 종류마다 어울리는 관리 방식이 다르다.

```text
UI 상태: 모달 열림, 선택된 탭
서버 상태: 게시글 목록, 사용자 정보
전역 상태: 로그인 사용자, 테마
폼 상태: 입력값, 검증 에러
```

각 상태의 성격에 맞는 관리 방식이 필요하다.

처음부터 모든 상태를 전역 store에 넣을 필요는 없다. 가까운 컴포넌트 안에서만 쓰이면 local state가 더 읽기 쉽고, 서버에서 가져온 데이터는 TanStack Query 같은 서버 상태 도구가 더 자연스럽다. Redux Toolkit이나 Zustand는 여러 화면이 같은 값을 공유하거나, 상태 변경 흐름을 따로 관리해야 할 때 힘을 발휘한다.

### 언제 별도 도구가 필요한가

아래 상황이면 상태 관리 도구를 고려한다.

- 여러 컴포넌트가 같은 값을 사용한다.
- props를 여러 단계로 계속 전달해야 한다.
- 상태 변경 로직이 복잡하다.
- 로그인 정보나 앱 설정처럼 전역 값이 있다.

<a id="redux-toolkit"></a>
## Redux Toolkit

Redux Toolkit은 Redux를 더 쉽게 쓰기 위한 공식 도구다.

Redux는 전역 상태를 하나의 store에서 관리한다.  
상태 변경은 action과 reducer를 통해 예측 가능하게 일어난다.

아래 상황에서 Redux Toolkit이 잘 맞는다.

- 전역 상태가 복잡하다.
- 상태 변경 흐름을 명확하게 추적하고 싶다.
- 큰 팀에서 일관된 패턴이 필요하다.
- Redux DevTools로 상태 변화를 디버깅하고 싶다.

간단한 앱에 Redux를 쓰면 코드가 필요 이상으로 많아질 수 있다.  
서버에서 가져온 데이터는 Redux보다 TanStack Query 같은 서버 상태 관리 도구가 더 적합할 때가 많다.

<a id="zustand"></a>
## Zustand

Zustand는 가볍고 단순한 React 상태 관리 라이브러리다.

간단한 store를 만들고 컴포넌트에서 필요한 상태만 가져다 쓴다.

```ts
const useStore = create((set) => ({
  count: 0,
  increase: () => set((state) => ({ count: state.count + 1 })),
}));
```

아래 상황에서 좋다.

- Redux보다 가벼운 전역 상태 관리가 필요하다.
- 보일러플레이트를 줄이고 싶다.
- 로그인 사용자, 테마, UI 상태처럼 작은 전역 상태가 있다.

너무 자유롭기 때문에 팀 규칙 없이 쓰면 store 구조가 흐트러질 수 있다.

<a id="pinia"></a>
## Pinia

Pinia는 Vue 생태계의 상태 관리 도구다.  
React의 Redux/Zustand처럼 Vue 앱에서 전역 상태를 관리할 때 사용한다.

Vue 프로젝트에서 아래 상황이면 Pinia를 사용한다.

- 여러 컴포넌트에서 같은 상태를 공유한다.
- 로그인 사용자나 설정 값을 전역으로 관리한다.
- Vue 공식 생태계에 가까운 상태 관리 방식을 쓰고 싶다.

<a id="frontend-testing"></a>
## Unit, Integration, E2E

프론트엔드 테스트는 보통 세 단계로 나눠 본다.

테스트는 "많이 작성하면 좋은 것"이라기보다, 실패하면 곤란한 흐름을 자동으로 확인하는 장치다.  
작은 계산이나 표시 로직은 Unit Test로, 여러 컴포넌트가 맞물리는 동작은 Integration Test로, 사용자가 실제로 밟는 흐름은 E2E Test로 확인한다.

### Unit Test

가장 작은 단위의 테스트다.  
함수, 컴포넌트 하나가 의도대로 동작하는지 본다.

### Integration Test

여러 조각이 함께 잘 동작하는지 본다.  
예를 들어 검색 입력창, 버튼, 결과 목록이 함께 동작하는지 확인한다.

### E2E Test

실제 사용자 흐름을 브라우저에서 테스트한다.

```text
로그인
-> 글쓰기
-> 게시글 상세 확인
-> 댓글 작성
```

중요한 사용자 흐름은 E2E로, 복잡한 로직은 Unit으로, 컴포넌트 조합은 Integration으로 테스트하는 식으로 나눠 생각하면 된다.

<a id="tailwind"></a>
## Tailwind

Tailwind CSS는 utility-first CSS 프레임워크다.  
미리 정의된 작은 클래스들을 조합해서 스타일을 만든다.

```tsx
<button className="rounded bg-black px-4 py-2 text-white">
  저장
</button>
```

아래 상황에서 좋다.

- CSS 파일을 많이 오가지 않고 빠르게 UI를 만들고 싶다.
- 디자인 시스템을 클래스 조합으로 일관되게 유지하고 싶다.
- 커스텀 UI를 직접 많이 만들고 싶다.

클래스가 길어질 수 있다.  
반복되는 스타일은 컴포넌트로 묶거나 유틸 함수로 정리하는 게 좋다.

<a id="mui"></a>
## MUI

MUI는 Material UI 기반의 React UI 컴포넌트 라이브러리다.

버튼, 모달, 테이블, 탭, 입력창 같은 완성된 컴포넌트를 제공한다.

아래 상황에서 좋다.

- 빠르게 관리자 페이지나 업무용 UI를 만들고 싶다.
- 접근성과 기본 상호작용이 구현된 컴포넌트를 쓰고 싶다.
- Material Design 스타일이 프로젝트와 잘 맞는다.

기본 스타일의 존재감이 강해서, 완전히 독자적인 디자인을 만들려면 커스터마이징 비용이 생길 수 있다.

<a id="shadcn"></a>
## shadcn

shadcn/ui는 복사해서 내 프로젝트 안에 소유하는 방식의 UI 컴포넌트 모음이다.  
일반적인 라이브러리처럼 패키지에서 컴포넌트를 import하는 방식과 조금 다르다.

Radix UI 기반 컴포넌트와 Tailwind 스타일을 조합해서 접근성 있는 UI를 빠르게 만들 수 있게 해준다.

아래 상황에서 좋다.

- Tailwind를 쓰고 있다.
- 컴포넌트 코드를 직접 수정하고 소유하고 싶다.
- 깔끔한 기본 UI를 빠르게 만들고 싶다.

컴포넌트 코드가 프로젝트 안에 들어오기 때문에, 업데이트 관리와 팀 스타일 규칙을 신경 써야 한다.

<a id="bootstrap"></a>
## Bootstrap

Bootstrap은 오래된 CSS/UI 프레임워크다.  
그리드 시스템, 버튼, 폼, 네비게이션 같은 기본 스타일을 제공한다.

아래 상황에서 좋다.

- 빠르게 기본적인 반응형 페이지를 만들고 싶다.
- 디자인 커스터마이징보다 안정적인 기본 UI가 중요하다.
- 레거시 프로젝트나 기존 Bootstrap 기반 프로젝트를 다룬다.

Bootstrap 느낌이 강하게 날 수 있다.  
요즘 React 프로젝트에서는 Tailwind, shadcn, MUI를 더 많이 선택하는 경우도 많다.

<a id="a11y"></a>
## a11y

a11y는 accessibility, 접근성을 줄여 쓴 표현이다.  
accessibility에서 a와 y 사이에 11글자가 있어서 a11y라고 부른다.

접근성은 모든 사용자가 웹을 사용할 수 있게 만드는 것이다.  
시각장애인, 키보드만 사용하는 사용자, 색상 구분이 어려운 사용자, 스크린 리더 사용자도 포함한다.

사실 모든 서비스에서 중요하다.  
특히 공공, 교육, 금융, 커머스처럼 많은 사용자가 접근하는 서비스에서는 필수에 가깝다.

- 버튼은 `button` 태그를 사용한다.
- 이미지에는 의미 있는 `alt`를 제공한다.
- 폼 입력에는 label을 연결한다.
- 키보드만으로 조작할 수 있어야 한다.
- 색상만으로 상태를 구분하지 않는다.
- 모달과 메뉴는 포커스 이동을 고려한다.

요약하면 접근성은 "더 많은 사람이 문제 없이 쓸 수 있게 만드는 품질 기준"이다.
