<a id="language-basics"></a>
## Language Basics

언어 기초는 특정 프레임워크를 쓰기 전에 알아야 하는 기본 체력이다.  
프레임워크는 계속 바뀌지만, 타입, 함수, 객체, 에러 처리, 테스트 같은 개념은 거의 모든 개발에서 반복해서 나온다.

내가 이 파트를 볼 때 중요한 기준은 하나다.

```text
"이 언어로 코드를 작성할 수 있나?"보다
"이 언어의 방식으로 문제를 구조화할 수 있나?"
```

예를 들어 Python을 쓴다고 해서 문법만 알면 되는 게 아니다. 비동기 함수, 예외 처리, 타입 힌트, 테스트 방식까지 알아야 서버 코드를 안정적으로 만들 수 있다. TypeScript도 마찬가지다. 타입 문법만 외우는 게 아니라 프론트엔드 상태와 API 응답을 타입으로 어떻게 표현할지 알아야 의미가 있다.

이 문서의 키워드들은 "어떤 언어가 더 좋다"를 비교하려는 게 아니다.  
오히려 각 언어가 개발자에게 어떤 습관을 요구하는지 보는 쪽에 가깝다. TypeScript는 데이터 모양을 먼저 생각하게 만들고, Java는 책임과 구조를 신경 쓰게 만들고, Python은 빠른 실험과 라이브러리 조합에 강하게 만든다.

<a id="language"></a>
## Language

언어는 사람이 컴퓨터에게 일을 시키기 위해 쓰는 표현 방식이다.  
하지만 실무에서 언어는 단순 문법이 아니라 생태계까지 포함한다.

언어를 고를 때는 보통 아래를 같이 본다.

- 어떤 문제를 잘 푸는가
- 어떤 라이브러리와 프레임워크가 많은가
- 팀원이 배우기 쉬운가
- 배포와 운영이 편한가
- 성능, 안정성, 생산성 중 무엇을 더 중시하는가

PDF에서는 TypeScript, Java, Python이 언어 키워드로 나온다. 세 언어 모두 웹 개발에서 많이 쓰이지만, 주로 맡는 역할이 조금 다르다.

<a id="typescript"></a>
## TypeScript

TypeScript는 JavaScript에 타입 시스템을 추가한 언어다.  
브라우저는 TypeScript를 직접 실행하지 못하기 때문에, 개발할 때 TypeScript로 쓰고 빌드 과정에서 JavaScript로 변환한다.

TypeScript는 코드가 실행되기 전에 많은 실수를 잡아준다.

```ts
function getUserName(user: { name: string }) {
  return user.name;
}

getUserName({ age: 20 }); // name이 없으므로 타입 에러
```

JavaScript였다면 실행 중에야 문제가 드러날 수 있지만, TypeScript는 개발 단계에서 바로 알려준다.

프론트엔드에서는 거의 표준처럼 쓰인다. React, Vue, Next.js 같은 현대 프론트엔드 프로젝트는 TypeScript를 쓰는 경우가 많다.  
백엔드에서도 Node.js/NestJS를 쓴다면 TypeScript가 많이 쓰인다.

특히 아래 상황에서 효과가 크다.

- API 응답 형태가 복잡하다.
- 컴포넌트에 전달되는 props가 많다.
- 여러 명이 같은 코드를 수정한다.
- 코드가 커져서 "이 값이 어떤 모양인지" 계속 헷갈린다.

### 초보자가 헷갈리기 쉬운 점

TypeScript는 런타임을 안전하게 만드는 도구가 아니다.  
예를 들어 서버에서 이상한 JSON을 보내면 TypeScript 타입만으로는 막을 수 없다. 런타임 검증이 필요하면 Zod, Valibot, Pydantic 같은 검증 도구를 함께 써야 한다.

요약하면 TypeScript는 "코드를 쓰는 동안 실수를 줄이는 언어"다.

<a id="java"></a>
## Java

Java는 오래된 만큼 생태계가 매우 크고, 대규모 백엔드 시스템에서 많이 쓰이는 언어다.  
Spring Boot와 함께 기업용 서버, 금융, 커머스, 공공 시스템에서 자주 사용된다.

Java는 정적 타입 언어다. 클래스, 인터페이스, 객체지향 설계가 강하게 자리 잡고 있다.

```java
public class User {
    private Long id;
    private String name;
}
```

Java의 장점은 안정성과 예측 가능성이다. 대규모 팀에서 오래 운영되는 서버를 만들 때 강하다.

아래 상황에서 Java가 많이 선택된다.

- 안정성이 중요한 대규모 백엔드
- Spring Boot 기반 서버
- 기업 환경에서 오래 유지보수해야 하는 시스템
- 명확한 계층 구조와 객체지향 설계가 필요한 시스템

### 초보자가 헷갈리기 쉬운 점

Java는 문법이 다소 장황하게 느껴질 수 있다. 하지만 장황함 덕분에 구조가 명확해지는 장점도 있다.  
요즘은 Lombok, record, Spring Boot 자동 설정 등으로 예전보다 훨씬 편하게 개발할 수 있다.

요약하면 Java는 "크고 오래 가는 서버 시스템에 강한 언어"다.

<a id="python"></a>
## Python

Python은 문법이 간결하고 읽기 쉬워서 빠르게 구현하기 좋은 언어다.  
웹 서버, 데이터 분석, 자동화, 머신러닝, AI 서비스에서 널리 쓰인다.

Python은 적은 코드로 많은 일을 하게 해준다.

```py
def get_user_name(user: dict) -> str:
    return user["name"]
```

FastAPI, Django, Flask 같은 웹 프레임워크가 있고, AI 쪽에서는 PyTorch, LangChain, LlamaIndex, OpenAI SDK 같은 라이브러리가 많다.

아래 상황에서 Python이 많이 선택된다.

- 빠르게 API 서버를 만들고 싶다.
- AI, 데이터 처리, 자동화를 함께 다루고 싶다.
- FastAPI로 문서화된 API를 빠르게 만들고 싶다.
- 복잡한 알고리즘보다 구현 속도와 라이브러리 활용이 중요하다.

### 초보자가 헷갈리기 쉬운 점

Python은 동적 타입 언어라서 처음에는 편하지만, 프로젝트가 커지면 타입이 헷갈릴 수 있다.  
그래서 요즘 Python 서버는 타입 힌트와 Pydantic 같은 검증 도구를 같이 쓰는 경우가 많다.

요약하면 Python은 "빠른 구현과 AI/데이터 생태계에 강한 언어"다.

<a id="paradigm"></a>
## Paradigm

프로그래밍 패러다임은 코드를 바라보는 방식이다.  
같은 문제라도 객체지향으로 풀 수도 있고, 함수형으로 풀 수도 있다.

패러다임은 언어와 완전히 같은 개념이 아니다. Java는 객체지향이 강하지만 함수형 스타일도 쓸 수 있고, JavaScript/TypeScript는 객체지향과 함수형 스타일을 모두 많이 쓴다. Python도 클래스와 함수를 둘 다 자연스럽게 쓴다.

<a id="oop"></a>
## OOP

OOP는 Object-Oriented Programming, 객체지향 프로그래밍이다.  
데이터와 그 데이터를 다루는 행동을 객체로 묶어서 생각한다.

예를 들어 게시글이라는 개념이 있다면, 게시글은 제목, 내용, 작성자 같은 데이터를 가진다. 그리고 수정하기, 삭제하기, 공개하기 같은 행동을 가질 수 있다.

```py
class Post:
    def __init__(self, title: str, content: str):
        self.title = title
        self.content = content

    def update_content(self, content: str):
        self.content = content
```

OOP는 복잡한 도메인을 구조화하는 데 도움을 준다.

아래 상황에서 객체지향이 잘 맞는다.

- 시스템 안에 명확한 개념이 많다.
- 도메인 규칙을 코드로 표현해야 한다.
- 여러 기능이 같은 데이터를 중심으로 움직인다.
- 코드가 커져서 역할과 책임을 나눠야 한다.

### 초보자가 헷갈리기 쉬운 점

객체지향은 무조건 클래스를 많이 만드는 것이 아니다.  
핵심은 책임을 적절히 나누는 것이다. "이 객체가 무엇을 알고, 무엇을 해야 하는가"를 생각해야 한다.

요약하면 OOP는 "현실 또는 도메인의 개념을 코드 객체로 모델링하는 방식"이다.

<a id="functional-programming"></a>
## Functional Programming

함수형 프로그래밍은 데이터를 바꾸기보다, 입력을 받아 결과를 반환하는 함수들의 조합으로 프로그램을 만드는 방식이다.

함수형 스타일에서는 가능하면 같은 입력에 대해 항상 같은 출력을 내는 순수 함수를 선호한다.

```ts
const addTax = (price: number) => price * 1.1;
const formatPrice = (price: number) => `${price.toLocaleString()}원`;

const result = formatPrice(addTax(10000));
```

데이터를 직접 바꾸기보다 새로운 값을 만들어내는 방식을 많이 쓴다.

아래 상황에서 함수형 스타일이 좋다.

- 데이터 변환이 많다.
- UI 상태를 예측 가능하게 관리하고 싶다.
- 테스트하기 쉬운 작은 함수를 만들고 싶다.
- 부작용을 줄이고 싶다.

프론트엔드에서 배열의 `map`, `filter`, `reduce`를 자주 쓰는 것도 함수형 스타일의 예시다.

### 초보자가 헷갈리기 쉬운 점

함수형 프로그래밍은 모든 것을 어렵게 수학적으로 만들자는 뜻이 아니다.  
초보 단계에서는 "입력과 출력을 명확히 하고, 불필요한 상태 변경을 줄인다" 정도로 이해해도 충분하다.

요약하면 함수형 프로그래밍은 "작고 예측 가능한 함수들을 조합하는 방식"이다.

<a id="error-handling"></a>
## Error Handling

에러 처리는 프로그램이 실패했을 때 어떻게 반응할지 정하는 것이다.  
개발에서 중요한 건 에러가 아예 안 나게 하는 것이 아니라, 에러가 났을 때 사용자가 망가지지 않게 하고 원인을 추적할 수 있게 하는 것이다.

에러 처리는 보통 아래를 포함한다.

- 잘못된 입력을 거절한다.
- 실패 원인을 사용자나 개발자에게 적절히 알려준다.
- 서버 내부 정보는 숨긴다.
- 로그를 남긴다.
- 재시도할 수 있는 실패와 재시도하면 안 되는 실패를 구분한다.

모든 코드에서 필요하다. 특히 API 서버에서는 더 중요하다.

예를 들어 로그인 API에서 비밀번호가 틀렸을 때와 DB가 죽었을 때는 완전히 다른 에러다.

```text
401 Unauthorized: 인증 실패
500 Internal Server Error: 서버 내부 문제
503 Service Unavailable: 일시적인 서비스 불가
```

### 초보자가 헷갈리기 쉬운 점

에러 처리를 `try-catch`로 감싸는 것만으로 생각하면 안 된다.  
좋은 에러 처리는 "실패를 분류하고, 적절히 표현하고, 기록하고, 회복할 수 있게 하는 설계"에 가깝다.

요약하면 에러 처리는 "실패를 시스템의 일부로 받아들이는 방법"이다.

<a id="test-framework"></a>
## Test Framework

테스트 프레임워크는 코드가 의도한 대로 동작하는지 자동으로 확인해주는 도구다.

테스트 프레임워크는 보통 아래 기능을 제공한다.

- 테스트 케이스 작성
- 실행 결과 검증
- 실패한 테스트 리포트 출력
- 테스트 전후 준비 작업
- mock, fixture 같은 보조 기능

예를 들어 Python에서는 pytest, Java에서는 JUnit, TypeScript에서는 Vitest/Jest가 많이 쓰인다.

아래 상황에서 특히 중요하다.

- 같은 기능을 여러 번 수정해야 한다.
- 리팩터링할 때 기존 기능이 깨졌는지 확인해야 한다.
- API 계약이 유지되는지 확인해야 한다.
- 버그가 다시 생기지 않도록 막고 싶다.

### 초보자가 헷갈리기 쉬운 점

테스트는 "완벽함을 보장하는 도구"가 아니다.  
하지만 중요한 동작을 자동으로 반복 확인해주기 때문에, 개발 속도와 안정성을 같이 올려준다.

요약하면 테스트 프레임워크는 "내 코드가 계속 같은 약속을 지키는지 확인하는 장치"다.
